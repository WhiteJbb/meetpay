# 의사결정 로그

주요 의사결정을 시간순으로 기록한다 (CLAUDE.md Working Rules 항목 5).
각 항목은 "무엇을 결정했는지"와 "왜 그렇게 결정했는지"를 포함한다.

---

## 2026-07-07

### 1. 서비스 정책 (10문10답 답변 기반, docs/PlanningDoc.md §2 참고)

- **부분 참여 방식**: 지출마다 참여자를 자유롭게 선택(B안).
  이유: 노래방처럼 일부만 참여하는 지출이 흔해 유연성이 더 중요하다고
  판단.
- **인증**: 회원가입 없이 모임 링크만으로 접근하는 익명 방식(A안).
  이유: 가입 절차가 있으면 참여자들이 이탈할 가능성이 커서, 진입 장벽을
  최소화하기로 함.
- **참여자 식별**: 실명 사용, 동명이인은 이메일로 구분.
  이유: 닉네임은 모임 내에서 혼동을 줄 수 있어 실명이 더 명확하다고
  판단, 단 동명이인 구분을 위해 이메일을 필수값으로 둠.
- **송금 연동**: 계산 결과만 제공하고 실제 송금은 사용자가 직접(A안).
  이유: 외부 결제/송금 API 연동은 공수와 리스크가 크므로 MVP 범위에서
  제외.
- **모바일/PC**: 처음부터 동등한 반응형 지원(C안).
  이유: 참여자마다 접속 환경이 다를 수 있어 특정 플랫폼에 치우치지
  않기로 함.
- **결과 공유**: 누구나 볼 수 있는 공유 링크 방식(B안).
  이유: 카카오톡 텍스트 생성 등은 이후 확장으로 미루고, 우선 링크 기반
  공유로 단순화.
- **통화**: 다국통화 지원(환율 변환 포함).
  이유: 해외여행 모임 사용 시나리오를 지원 범위에 포함하기로 함.
- **영수증 첨부**: 첨부 가능하지만 선택 사항(B안).
  이유: 필수로 강제하면 사용성이 떨어지고, 미지원하면 증빙 니즈를
  놓치므로 절충.
- **모임 규모**: 소규모(2~10명, 지출 수십 건) 위주로 최적화.
  이유: 주 사용처가 친구/소모임이라고 판단, 대규모 모임 성능 최적화는
  범위 밖으로 둠.
- **이력 보관**: 정산 완료 후 일정 기간(90일)이 지나면 자동 삭제.
  이유: 개인정보(실명, 이메일)를 무기한 보관하는 리스크를 줄이기 위해.

### 2. 기술 스택: Next.js + TypeScript + SQLite (Prisma)

사용자에게 3가지 안(Next.js 풀스택 / Express+React+Postgres / 직접 지정)을
제시했고, 사용자가 "Next.js + TypeScript + SQLite"를 선택.
이유(제안 시 근거): 별도 서버/DB 인프라 없이 로컬에서 빠르게 MVP를
구축할 수 있고, 소규모 모임 대상 서비스 특성상 SQLite로 충분하다고
판단.

### 3. Prisma 7 드라이버 어댑터 도입 (@prisma/adapter-better-sqlite3)

Prisma 7부터 `PrismaClient` 생성 시 드라이버 어댑터가 필수가 되어(구버전
방식인 datasource url 직접 연결 불가), SQLite용
`@prisma/adapter-better-sqlite3`를 도입. 이유: 설치된 Prisma 버전이
7.8.0이라 어댑터 없이는 타입 에러 발생(자세한 경과는
`docs/WorkLog.md` 참고).

### 4. 환율 처리: 실시간 API 대신 고정 참고 환율 사용 (MVP)

`src/lib/currency.ts`에 KRW 기준 고정 환율 테이블을 두고 사용. 이유:
실시간 환율 API 연동처(공급자 선택, 캐싱, 장애 대응 등)는 결정이 더
필요한 사항이라 MVP 범위에서는 고정값으로 단순화하고, 이후 확장
범위로 문서화(`docs/PlanningDoc.md` §6).

### 5. 영수증 첨부: 로컬 파일 업로드 방식

외부 스토리지(S3 등) 연동 없이 `public/uploads/receipts`에 저장하는
방식으로 구현. 이유: 별도 스토리지 계정/인프라 결정 없이도 "첨부 가능"
요구사항을 실제로 동작하게 만들기 위한 MVP 수준 구현. 프로덕션 배포 시
영구 스토리지로 교체가 필요함을 인지하고 있음.

### 6. 이력 삭제: 접근 시점 지연 삭제(lazy delete) 방식

정해진 배치/크론 작업 대신, 모임 페이지 접근 시 만료 여부를 확인해
그 자리에서 삭제하는 방식(`src/lib/group.ts`)으로 구현. 이유: 로컬
개발 환경에는 크론 인프라가 없어, MVP 단계에서는 접근 시점 검사로
대체하고 실제 배포 시 정기 배치로 보완 필요함을 남겨둠.

### 7. 페르소나 검토 결과 전체 반영 — IA 개정 (사용자 승인)

페르소나 서브에이전트 2회차 검토(`docs/ReviewChecklist.md` "페르소나
IA 검토 결과")에서 나온 개선 후보 전체를 IA에 반영하기로 사용자가
승인함("세명이 불편했던 지점들 다 반영해서 IA 업데이트해"). Approval
Gate 중 **IA 변경**, **개인정보 필드 추가(계좌)**, **정책 변경(이메일
필수 완화)**에 해당하는 결정.

- **링크 체계 확정**: 모임 링크(편집 가능 → 모임 홈 진입) / 결과
  링크(열람 전용 → 정산 결과 직행) 2종으로 확정하고, 별도의 "결과
  공유 링크(읽기 전용)" 화면은 정산 결과 화면의 읽기 전용 모드로
  통합. 이유: 3명 모두 링크/화면 이원화를 혼란 지점으로 지적했고,
  링크=권한인 익명 서비스에서 편집 링크 오배포 리스크를 줄이기 위해.
- **"정산하기" → "정산 결과 보기"로 버튼명 변경**: 마감이 아닌 조회
  액션임을 명확히 하고, 만료 기산과 무관함을 IA에 명시. 이유: 중간
  점검을 주저하게 만드는 어감(지현), 관리 액션처럼 보여 열람자가
  누르기를 망설이는 문제(수아).
- **이메일 필수 → 조건부 필수로 완화**: 이름만으로 등록 가능, 동명이인
  발생 시에만 이메일 필수. 이유: "1분 내 모임 생성" 성공 기준의 최대
  걸림돌(지현). 10문10답 #3의 원 취지(동명이인 구분 수단 = 이메일)는
  유지하면서 상시 필수라는 구현 방침만 완화.
- **계좌 정보 선택 입력 추가(개인정보 필드)**: 참여자 등록/관리에서
  선택 입력, 정산 결과 이체 목록 각 줄에 받는 사람 계좌 표시+복사.
  이유: UserScenarios 시나리오 3("계좌 복사 → 송금")과 10문10답 #4
  ("송금은 사용자가 직접, 계좌 복사 등")가 전제하는 기능인데 IA에
  입력/표시 지점이 없어 문서 간 모순 상태였음. 링크 뒤 노출 + 만료 시
  삭제 원칙 적용. **구현 시 DB 스키마 변경이 필요하며, 이는 Plan.md
  갱신 시점에 별도 확인 후 진행한다.**
- **기준 통화는 KRW 고정 유지, 표시만 추가**: 모임 생성/홈에 "KRW
  기준 환산" 안내 표시, 정산 결과에 적용 환율 표("1 JPY = 9 KRW,
  고정 참고 환율") 명시. 이유: 기준 통화 개념이 정산 결과에서 갑자기
  등장하는 혼란(민수) 해소. 기준 통화의 사용자 선택 기능은 이번에
  도입하지 않음(별도 정책 결정 필요, 필요 시 이후 논의).
- **그 외 화면 구성 보강**: 참여자 추가/수정/삭제(모임 홈 내 섹션),
  지출 삭제 버튼, 지출 총액·순 잔액 요약(모임 홈), 환산 미리보기,
  지출일 기본값 오늘, 저장 후 계속 추가, 영수증 첨부 아이콘·열람 경로,
  내 항목 강조, 정산 결과 → 모임 홈 복귀 경로. 이유: 각 페르소나의
  성공 기준(1분 생성, 계산 실수 방지, 환산 신뢰, 원클릭 결과 확인)을
  IA 수준에서 보장하기 위해.

일관성을 위해 `docs/PlanningDoc.md`(4.1 참여자 등록, §3 핵심 기능,
§5 보안)와 `docs/UserScenarios.md`(시나리오 1·2의 버튼/링크 명칭)도
함께 갱신함. 코드 구현은 아직 진행하지 않음 — 구현 착수 전
`docs/Plan.md` 갱신 필요. (→ §8에서 구현 진행)

### 8. IA 개정분 구현 — 스키마 변경 및 구현 세부 결정

§7 반영 후 "계좌 정보는 DB 스키마 변경이 필요하다"고 고지한 상태에서
사용자가 "구현해줘"로 지시 → 스키마 변경을 포함한 구현 승인으로 보고
진행(Approval Gate: DB 스키마 변경). 구현 중 내린 세부 결정:

- **스키마 변경(비파괴 마이그레이션)**: `Participant.email`을
  선택(`String?`)으로 완화, `Participant.bankAccount`(`String?`) 추가,
  `Group.viewToken`(`String? @unique`) 추가. viewToken을 필수가 아닌
  선택 필드로 두고 기존 행은 접근 시점에 지연 발급 — 기존 데이터
  리셋 없이 마이그레이션하기 위함(Approval Gate "데이터 삭제" 회피).
- **계좌 필드는 자유 입력 한 칸**: 은행/계좌번호를 분리하지 않고
  "국민 123-456" 같은 자유 텍스트 하나로 받음. 이유: 표시/복사
  용도뿐이라 구조화가 불필요하고 입력 마찰이 적음.
- **참여자 삭제 정책(PlanningDoc 4.1의 미결 사항 확정)**: 지출에
  연관된(지출자이거나 분담자인) 참여자는 삭제 불가로 차단하고 사유를
  UI에 표시. 참여자가 2명이 되면 추가 삭제 불가. 이유: 지출 이력을
  둔 채 참여자를 지우면 정산 정합성이 깨지므로 보수적 기본값을 택함.
  (지출 재배정 후 삭제 같은 흐름은 이후 필요 시 논의)
- **동명이인 검증은 앱 레벨**: 같은 이름이 2명 이상이면 각자 서로
  다른 이메일을 요구하는 검증을 모임 생성/참여자 추가·수정 액션에서
  수행. DB 제약은 `@@unique([groupId, email])` 유지.
- **지출일은 UTC 자정으로 저장**: 화면 표시가 `toISOString()` 기반이라
  로컬 자정으로 저장하면 KST에서 하루 밀려 보이는 버그가 있어(검증
  중 발견) UTC 자정으로 통일.

### 9. 전체 화면 비주얼 디자인 (design-boost)

사용자 지시("화면 예쁘게 만들어줘")로 전체 화면에 디자인 시스템을
새로 적용. 이전에는 Tailwind 기본 neutral 팔레트 + Geist 폰트(한글
미지원, 시스템 폰트로 폴백)만 사용하던 상태.

- **콘셉트: "정산 장부(영수증)"** — 이 서비스의 본질(모임 지출을
  적고 최소 이체로 정산)을 소재로 삼음. 장부지 색 배경 위에 흰
  영수증 카드가 놓이는 레이아웃, 정산 요약과 이체 목록 사이에
  절취선(점선+리벳 점) 하나만 시그니처로 사용.
- **색**: bg `#EFEBE0`(장부지 회색) · text `#22201A`(잉크 남흑) ·
  accent `#8C6415`(황동/도장 잉크, 흰 글자 대비 5.33:1 확보) ·
  참여자 잔액의 +/-는 accent와 분리된 고정 의미색
  positive `#2F7A4D` / negative `#B23B3B` 사용(크림+세리프+테라코타
  같은 뻔한 조합, 보라/파랑 그라디언트 히어로 등 흔한 패턴 회피).
- **폰트**: 제목 IBM Plex Sans KR, 본문 Gowun Dodum, 금액/날짜는
  IBM Plex Mono(자릿수 정렬) — next/font/google로 로드, subsets는
  해당 한글 폰트들의 유일한 등록 서브셋인 "latin"만 지정해도 한글
  글리프가 정상 렌더링됨(로컬 font-data.json으로 확인 후 결정).
- **토큰화**: Tailwind v4 `@theme inline`에 색·라운드·그림자를
  전부 정의해 `bg-bg`/`text-ink`/`bg-accent`/`rounded-token`/
  `shadow-card` 등 유틸리티로 생성 — 컴포넌트에 임의 색상 값 없음.
- **모션**: 이체 목록 항목에만 스태거 페이드인(정산 결과가 드러나는
  유일한 모션), `prefers-reduced-motion` 대응.
- **범위에서 제외한 것**: 다크 모드(기존에도 동작하지 않던 죽은 코드
  였고 사용자 요청도 없어 라이트 모드에 집중).
- **검증**: `tsc --noEmit`/`next build` 통과, Playwright로 데스크톱
  +모바일(390px) 스크린샷 촬영 후 오버플로/겹침 없음 확인, 라벨
  텍스트 색이 디자인 의도(잉크색)대로 렌더링되는지 `getComputedStyle`
  로 재확인(스크린샷 압축으로 인한 시각적 착시였음, 실제 색상은
  `rgb(34,32,26)`로 정상).

### 10. 디자인 방향 전환 — "정산 장부(영수증)" → 클린 카드 기반 디자인 시스템

사용자가 `reference/` 폴더에 3장의 레퍼런스 이미지(각 파일명에 마음에
드는/아쉬운 지점을 직접 기록)를 두고 무드보드·브랜드 키워드 정리와
`docs/DesignSystem.html` 제작을 지시함. Approval Gate "디자인 방향
변경"에 해당하나, 사용자가 레퍼런스를 직접 큐레이션해 지시한 것 자체가
방향 승인이라고 보고 진행. §9의 "정산 장부" 콘셉트(베이지·잉크·황동,
세리프+한글폰트 3종, 절취선 모티프)를 대체하는 **새 디자인 시스템

### 11. 참여자 아바타 색 시스템 도입 — 토큰 교체만으로는 "비슷해 보임" 문제 해결

앱 전체 리스킨(§10 적용, `docs/Plan.md` "DesignSystem.html 반영") 직후
사용자가 "디자인시스템 전체 적용해야지 왜 디자인이 거의 비슷해?"라고
지적. 원인: 색·라운드·폰트 **토큰만** 교체하고 레이아웃/컴포넌트
형태는 손대지 않아 구조적으로 이전과 동일해 보였음. 레퍼런스
3장에서 가장 강한 공통 신호가 "사람마다 다른 색의 원형 아바타"였는데
반영이 빠져 있었던 것이 핵심 원인.

- **참여자 아바타 색 팔레트 신설**(`src/lib/avatarColor.ts`): sky/
  peach/lavender/rose/teal/sand 6색. accent(그린)는 CTA·잔액 의미색
  전용으로 남기고, 아바타는 "사람 구분"이라는 별도 목적의 정체성
  색으로 분리 — 의미 충돌 방지(§10 Do/Don't "accent를 잔액 외
  의미에 쓰지 않는다"와 일관).
  - `avatarColorForName(name)`: 이름 문자열 해시 → 팔레트 인덱스.
    같은 사람은 참여자 목록/지출 내역/정산 결과 전체에서 항상 같은
    색(참여자 목록·지출 결제자·잔액 목록·이체 목록·랜딩 미리보기에
    일관 적용).
  - `avatarColorForIndex(i)`: 모임 생성 폼처럼 아직 이름을 타이핑
    중이라 값이 비어 있을 수 있는 행에서 색이 keystroke마다 바뀌지
    않도록 행 순서 기반 고정색 사용.
- **공용 `Avatar` 컴포넌트**(`src/components/Avatar.tsx`, 서버/클라
  양쪽에서 재사용): `groups/[id]`(참여자·지출 목록), `ExpenseForm`
  (분담 참여자 체크박스 → 아바타가 붙은 pill 칩으로 전환),
  `SettlementView`(잔액 목록), `TransferList`(이체 행·내 이름 선택
  칩), 랜딩 미리보기 카드, `groups/new`(입력 중 실시간 아바타
  미리보기)에 전부 적용.
- **액션 링크 → pill 버튼 전환**: "수정"/"삭제"가 밑줄 텍스트
  링크였던 것을 배경이 있는 작은 pill 버튼으로 바꿔 "기본 브라우저
  링크처럼 보인다"는 인상을 제거(`ConfirmSubmitButton` 기본
  className, `groups/[id]` 참여자·지출 목록의 수정/삭제).
- **헤더 브랜드 마크 추가**: `layout.tsx` 상단 "정산" 워드마크 앞에
  그린 accent 원형 배지(₩)를 추가 — 브랜드 아이덴티티가 텍스트뿐이던
  것을 보완.
- **지출 총액 카드를 다크 스탯 카드로 전환**: `groups/[id]`의 "지출
  총액" 카드를 `bg-ink`(다크) 배경 + 큰 모노스페이스 금액으로 변경—
  DesignSystem.html의 Stat Card 패턴을 실제로 가져와 화이트 카드
  일색이던 화면에 위계를 추가.

  전체(랜딩→모임 생성→지출 추가→모임 홈→정산 결과) 데스크톱/

### 12. DesignSystem.html 컴포넌트 타입 실제 도입 — "형식이 그대로"라는 재지적 반영

§11 이후에도 사용자가 "design system 적용해서 전면개편하라니까? 왜
자꾸 전 형식을 유지하는거야"라고 재차 지적. §10~§11은 색·아바타 등
"표면" 토큰만 바꿨을 뿐, §9에서 확정한 화면 골격(각 페이지 상단 h1+
설명 한 줄, 참여자/지출목록을 한 페이지에 쭉 나열, `window.confirm`
삭제, 기본 Next.js 404)은 그대로였음 — 즉 "형식"(레이아웃/상호작용
구조) 자체는 안 바뀐 것이 재지적의 핵심. 이번엔 §10 문서화 당시
"범위 밖으로 남긴 것"으로 미뤘던 컴포넌트 타입(Modal/Tabs/Navigation
/Group Card/Empty State/Permission Denied/Loading State)을 실제
화면에 도입.

- **AppBar**(`src/components/AppBar.tsx`, DesignSystem "8. 내비게이션"):
  각 페이지 상단의 `<h1>` 단독 배치를 뒤로가기 버튼+제목+부제목이
  있는 카드형 앱바로 교체. `groups/new`, 지출 추가/수정,
  `SettlementView`에 적용(모임 홈은 GroupCard가 그 역할을 겸함).
- **GroupCard**(`src/components/GroupCard.tsx`, "10. 모임 카드"):
  모임 홈의 흩어져 있던 제목/부제/총액 박스/링크 복사 버튼을 만료
  D-day 배지+통계 3열(총액/참여자/지출건수)+링크 복사가 한 카드에
  담긴 단일 컴포넌트로 통합.
- **Tabs**(`src/components/Tabs.tsx`, "9. 탭"): 모임 홈의 "참여자"
  섹션과 "지출 목록" 섹션을 세로로 쭉 나열하던 것을 밑줄 탭 전환으로
  변경 — 페이지 길이를 줄이고 DesignSystem의 탭 패턴을 실제로
  체감되게 함. (Route/IA 변경 아님 — 같은 페이지 안에서의 표시 방식
  변경이라 Approval Gate "IA 변경"에는 해당하지 않는다고 판단)
- **ConfirmDialog**(`src/components/ConfirmDialog.tsx`, "7. 모달"):
  참여자 삭제·지출 삭제에 쓰이던 `window.confirm()`을 네이티브
  `<dialog>` 기반 실제 모달로 교체(제목/본문/취소·삭제 버튼, 배경
  딤 처리). 브라우저 알림창이라 디자인 시스템을 벗어나 있던 지점을
  없앰.
- **EmptyState**(`src/components/EmptyState.tsx`, "15. 빈 상태"):
  "아직 등록된 지출이 없어요" 한 줄 텍스트를 아이콘+제목+설명+CTA
  버튼 카드로 교체.
- **Permission Denied**(`src/app/not-found.tsx`, "17. 접근 권한
  없음"): `notFound()` 호출 시 기본 Next.js 404 화면이 뜨던 것을
  자물쇠 아이콘+"링크가 만료되었거나 잘못된 주소예요"+90일 보관
  안내+홈 버튼으로 교체. 지금까지 완전히 미디자인 상태였던 화면이라
  체감 차이가 가장 컸음.
- **Loading State**(`src/app/groups/[id]/loading.tsx`, "14. 로딩
  상태"): 모임 홈 데이터 로딩 중 실제 레이아웃과 같은 크기의 shimmer
  스켈레톤을 보여주는 Next.js `loading.tsx` 세그먼트 추가.
  `globals.css`에 `.skeleton` shimmer 애니메이션 추가
  (`prefers-reduced-motion` 대응).
- **범위에서 계속 제외한 것**: DesignSystem의 "Tables"(데스크톱 표
  형태)는 이번에도 미적용 — 카드/리스트 UI와 중복 구현 비용 대비
  체감 효과가 상대적으로 낮다고 판단, 사용자가 원하면 후속 작업으로.
- **검증**: `tsc --noEmit`/`next build`(캐시 삭제 후) 통과. Playwright로
  골든 패스 전체 재구동 + 참여자 탭 전환, 삭제 확인 모달 열림
  (`dialog[open]` 확인), 존재하지 않는 모임 URL 접근 시 Permission
  Denied 화면 렌더 확인. 콘솔 에러 0건.

### 13. 색 토큰 절반이 실제 CSS 변수로 존재하지 않던 문제 수정

사용자가 스크린샷과 함께 "페이지 배경이 이 색이 아니지 않니? 디자인
시스템에서 색상 다 적용해"라고 재차 지적. 실제 배경(`--bg
#F6F5F0`)은 `getComputedStyle`로 정상 적용 확인됐지만, 근본 문제는
따로 있었음: `docs/DesignSystem.html`에는 정의돼 있는 `surface-alt`,
`accent-soft`, `warn`/`warn-soft`, `negative-soft` 토큰이 실제
`globals.css`(`@theme inline`)에는 **한 번도 추가된 적이 없었고**,
그 자리마다 `color-mix(in srgb, var(--accent) 12%, var(--surface))`
같은 인라인 임시값으로 대체해왔음. 즉 사용자가 "색상 다 적용"이라고
표현한 것이 정확히 맞는 지적 — 토큰 체계의 절반만 실제로 존재했음.

- `globals.css`에 5개 토큰(`--surface-alt`, `--accent-soft`,
  `--negative-soft`, `--warn`, `--warn-soft`)을 `@theme inline`에
  정식 추가(`bg-surface-alt`/`bg-accent-soft`/`text-warn` 등 Tailwind
  유틸리티로 생성됨).
- 앱 전체의 인라인 `color-mix()`/임의값(`bg-[color-mix(...)]`) 11곳을
  전부 제거하고 새 토큰 유틸리티로 교체(`groups/new`, `expenses/new`,
  `groups/[id]`, `not-found.tsx`, `AppBar`, `ConfirmDialog`,
  `ExpenseForm`, `ExpiryBanner`, `GroupCard`, `SettlementView`).
  모달 배경 딤 처리(`backdrop:bg-[color-mix(in_srgb,var(--ink)_35%,transparent)]`)
  1곳만 알파 블렌드 목적의 정상적인 예외로 남김.
- **부수 발견**: `GroupCard`의 만료 배지가 accent(그린) 색으로 잘못
  칠해져 있었음 — DesignSystem.html 원본 스펙은 만료 배지를
  `warn`/`warn-soft`로 정의하는데, `warn` 토큰 자체가 없었어서
  구현 시 대체재로 accent를 썼던 것. 토큰 추가 후 warn으로 수정.
- 미사용으로 남아있던 `ConfirmSubmitButton.tsx`(§12에서 `ConfirmDialog`로
  전량 교체된 뒤 죽은 코드였음) 삭제.
- **운영 사고**: 검증 중 `next build`를 다시 돌렸다가 `.next` 캐시에
  옛 CSS가 남아, 방금 켜져 있던 사용자의 개발 서버가 새 토큰 클래스를
  전혀 반영하지 못하는 상태가 됨(§10에서 이미 한 번 겪은 함정에
  재차 걸림). 서버를 재기동하고 `.next`를 삭제해 복구, 컴파일된 CSS에
  `.bg-warn-soft`/`.bg-accent-soft` 등 규칙이 실제로 존재하는지
  파일 레벨로 직접 확인 후 스크린샷으로 재검증.
- **검증**: `tsc --noEmit`/`next build` 통과. 만료 배지
  `getComputedStyle` 확인(`background: rgb(251,243,218)`,
  `color: rgb(185,137,0)` = `#FBF3DA`/`#B98900`, 토큰값과 정확히
  일치). Playwright 골든 패스 재구동, 콘솔 에러 0건.

### 14. 랜딩 히어로에 풀블리드 컬러 블록 도입 — "이게 적용된 걸로 보이냐" 재지적

색 토큰 수정(§13) 이후에도 사용자가 넓은 화면 스크린샷과 함께
같은 취지로 재차 지적. 토큰 자체는 정확했지만(배경 `#F6F5F0` 정상),
1920px 같은 넓은 데스크톱 화면에서 보면 콘텐츠는 가운데 640px
칼럼에만 있고 나머지는 전부 같은 베이지색 빈 공간이라 "디자인이
아예 안 된 페이지"처럼 보였음 — 톤이 아니라 **레이아웃의 시각적
공백**이 근본 원인.

- 랜딩 페이지(`src/app/page.tsx`) 최상단에 풀블리드(전체 뷰포트 폭)
  accent 그린 히어로 배너 추가. `layout.tsx`가 모든 페이지를
  `max-w-[640px]` 중앙 정렬 컨테이너로 감싸는 구조라, CSS
  `margin-left/right: calc(50% - 50vw)` breakout 기법으로 그 제약을
  뚫고 뷰포트 전체 폭까지 확장 — 내부 콘텐츠는 다시 640px 칼럼으로
  정렬해 다른 섹션과 정렬을 맞춤.
  이유: 무드보드 레퍼런스 1(민트/다크 대비 히어로)에서 채택하기로
  했던 "과감한 색 블록" 무드가 실제로는 랜딩 어디에도 적용되지
  않았던 것을 뒤늦게 인지 — 폼/카드 위주 화면(모임 생성, 지출 추가
  등)은 계속 절제된 톤을 유지하되, 첫인상인 랜딩 히어로만큼은 색을
  과감하게 써서 "디자인이 적용됐다"는 인상을 명확히 주기로 함.
- 히어로 배경은 accent, 텍스트는 accent-ink(어두운 그린)로 대비
  확보, CTA 버튼은 ink(잉크 다크)를 써서 그린 배경 위에서도 별도로
  튀도록 구성(기존 페이지의 accent 버튼과는 다른 예외적 처리 —
  히어로 배경 자체가 이미 accent이므로).
- **범위**: 이번엔 랜딩 페이지 한 곳만 수정. 모임 생성/지출 추가/
  모임 홈 등 폼·카드 중심 화면은 데스크톱에서도 여전히 640px 중앙
  칼럼을 유지(레퍼런스가 모두 모바일 앱 스크린샷이라 좁은 카드형
  레이아웃이 그 화면들에는 더 맞는다고 판단) — 필요 시 후속 지시로
  확장.
- **검증**: `tsc --noEmit`/`next build`(§13 함정 재확인 차 캐시
  삭제 후 재기동) 통과. Playwright로 1920px/390px 양쪽 스크린샷,
  breakout 기법이 가로 스크롤을 만들지 않는지
  `document.documentElement.scrollWidth === clientWidth`로 확인,
  콘솔 에러 0건.

### 15. 배경색 재변경 — 흰색과 확실히 구분되는 톤 + 히어로를 레퍼런스 2(블랙×네온) 무드로 전환

§13~§14에서 배경색이 정확히 적용됐음을 픽셀 단위로 증명했음에도
사용자가 "레퍼런스 참고해서 메인 디자인 좀 바꿔주고 배경색도
바꿔줘 마음에 안들어"라며 값 자체를 바꿔달라고 요청. 기술적으로는
맞았지만 `#F6F5F0`가 흰색과 육안으로 거의 구분되지 않을 만큼
미묘한 톤이었다는 사용자 체감을 인정하고, 더 뚜렷하게 색이
있다는 게 느껴지는 톤으로 교체.

- `--bg`: `#F6F5F0`(웜 그레이) → `#E3F0E9`(세이지 민트) — 흰색과
  나란히 놓아도 명확히 구분되는 채도로 변경.
- `--accent-soft`: `#E3F7EC` → `#D3ECDF`로 살짝 진하게 조정 — 새
  `--bg`와 값이 너무 비슷해져 카드 위 강조 배너(참여자 안내,
  ExpiryBanner 등)가 배경에 묻히는 걸 방지하기 위한 위계 재조정.
- `--border`: `#E3E0D5`(웜 탄색) → `#CFE2D7`(세이지 톤 보더)로
  변경해 새 배경 색상군과 어울리도록 조정.
- **랜딩 히어로를 레퍼런스 1(민트/다크) → 레퍼런스 2(블랙×네온그린)
  무드로 전환**: 배경을 `bg-accent`에서 `bg-ink`(다크)로 바꾸고,
  헤드라인은 밝은 `text-bg`, 강조 구간("링크 하나로")만
  `text-accent`로 네온처럼 튀게 처리. CTA 버튼은 계속 accent 배경 +
  accent-ink 텍스트로 어두운 배경 위에서 확실히 튀도록 유지.
  이유: 레퍼런스 2(Cash App류)의 블랙×네온 대비가 3장 중 가장
  뚜렷하고 "이건 확실히 디자인됐다"는 인상을 주는 조합이라 판단.
- **검증**: 새 배경색을 `elementFromPoint` + `getComputedStyle`로
  재확인(`rgb(227,240,233)` = `#E3F0E9`, 마진 영역에서 정확히
  일치). `tsc --noEmit`/`next build` 통과. `next build` 실행 직후
  곧바로 `.next` 삭제 + dev 서버 재기동(§13 캐시 함정 재발 방지
  루틴을 이번엔 선제적으로 적용). Playwright로 1920px/390px 양쪽
  랜딩 + 모임 생성 + 모임 홈 스크린샷, 콘솔 에러 0건.

### 16. 라이트 테마 → 다크 테마로 전면 전환 — "다크모드로 보고 있었다" 재구성

§13~§15에서 배경색이 정확히 적용됐음을 여러 차례 증명했음에도
불일치가 계속되자, 사용자가 `docs/DesignSystem.html`의 색상 팔레트
섹션 스크린샷을 보내며 "코드 기준 말고 저 보이는 색상 기준으로
맞춰달라"고 요청. 스크린샷을 분석해 근본 원인을 발견: 사용자의
브라우저/OS가 다크 모드였고, `DesignSystem.html`은 §10에서
`prefers-color-scheme: dark`를 지원하도록 만들어뒀기 때문에 문서가
다크 팔레트로 렌더되고 있었음. 그런데 스와치 카드의 **hex 텍스트
라벨은 라이트모드 값을 하드코딩**해뒀던 반면 **스와치 색상 자체는
`var()`를 참조해 다크모드 값으로 바뀌어 있어서**, 표시된 색상 칩과
그 아래 적힌 hex 코드가 서로 안 맞는 상태로 보였던 것. 이 라벨/실제
색 불일치가 오늘 하루 종일 반복된 "배경색이 이상하다" 지적의
실제 근본 원인이었을 가능성이 큼 — 사용자는 계속 다크모드로 문서를
보고 있었고, 실제 앱(라이트 전용, 다크모드 미지원)과 비교하면서
불일치를 느꼈을 것으로 추정.

- **결정**: `docs/DesignSystem.html`의 다크모드 스와치 값을 앱의
  **유일한 기본 테마**로 채택(별도 다크모드 토글 없이 이 값 자체가
  `:root`가 됨). §9에서 "다크모드는 범위 제외"로 결정했던 것을
  뒤집는 조치 — 사용자가 명시적으로 다크 팔레트를 요청했고, 지금까지
  실제로 이 팔레트를 보고 피드백해왔다는 정황도 확인됨.
- 새 토큰: `bg #14170F`, `surface #1E231A`, `surface-alt #262C21`,
  `ink #EDEFE7`(밝은 텍스트), `muted #A6AC9C`, `accent #23D67F`,
  `accent-hover #4FE39A`, `accent-soft #17321F`, `border #333A2C`,
  `negative #F0696E`, `negative-soft #3A2222`, `warn #E0B23D`,
  `warn-soft #3A331A`. **`accent-ink`만 문서의 다크모드 값
  (`#B8F5D2`, 밝은 민트)을 그대로 쓰지 않고 `#06301D`(어두운
  그린)로 보정** — 밝은 accent(`#23D67F`) 위에 밝은 텍스트를 얹으면
  대비가 부족해 원래 라이트모드에서 검증했던 어두운 텍스트 조합을
  유지.
- **`ink`/`bg` 토큰 의미 반전에 따른 연쇄 수정**: 라이트 테마에서는
  `ink`=어두운 텍스트, `bg`=밝은 배경이었지만 다크 테마에서는 정반대
  (`ink`=밝은 텍스트, `bg`=어두운 배경)가 되므로, 이 두 토큰을
  "라이트 전용 방향"으로 리터럴하게 썼던 곳을 재점검:
  - 랜딩 히어로(`page.tsx`): 페이지 전체가 이미 다크 배경이 됐으므로
    "다크 밴드"를 별도로 만들 필요가 없어짐 — 히어로를 `bg-accent`
    (네온 그린 블록)로 바꾸고 CTA 버튼을 `bg-ink`(다크 테마에서는
    밝은 색)로 반전, 그린 배경 위에서 흰 필 버튼이 튀도록 재구성.
  - `ConfirmDialog`의 모달 배경 딤 처리가 `var(--ink)` 35% 믹스를
    썼던 것을 발견 — `ink`가 밝은 색으로 바뀌면서 딤 처리가 오히려
    화면을 밝히는 역효과가 났을 것. 테마와 무관하게 항상 어둡게
    깔리도록 `bg-black/50`으로 고정.
- **검증**: `tsc --noEmit`/`next build` 통과, 빌드 직후 `.next` 삭제
  + 서버 재기동. Playwright로 랜딩(1920px/390px)·모임 생성·모임
  홈·참여자 탭·지출 추가·정산 결과 전 구간 스크린샷, 배경색
  `elementFromPoint` 재확인(`rgb(20,23,15)` = `#14170F` 정확히
  일치), 콘솔 에러 0건.

### 17. 전역 헤더 제거 — 랜딩 헤드라인을 첫 화면의 유일한 주인공으로

사용자가 "위에 header? 저거 정산 저거 없애버리고 메인
'더치페이...' 이걸 강조하는게 나을 거 같아. 밑에 모임 예시 저것도
디자인 수정 좀 필요할 거 같고"라고 요청.

- `layout.tsx`의 전역 `<header>`(모든 페이지 상단의 "정산" 워드마크
  바) 완전 제거. 대신 브랜드 마크(₩ 배지 + "정산")는 랜딩 히어로
  안쪽에 작게 접어 넣어 브랜드 아이덴티티 자체는 유지.
  - **영향 검토**: 하위 페이지들은 이미 `AppBar` 컴포넌트(§12)가
    자체적으로 뒤로가기/제목을 제공하므로 전역 헤더 없이도 내비게이션
    공백이 생기지 않음. 모임 홈(`groups/[id]`)에서 랜딩으로 돌아가는
    링크만 없어지는데, 이 앱은 링크 기반 접근 모델(IA.md §1)이라
    모임 안에서 마케팅 랜딩으로 돌아갈 일이 거의 없다고 판단해 범위
    안에서 수용.
- 랜딩 히어로(`page.tsx`)를 뷰포트 맨 위까지 채우도록 `marginTop:
  -2.5rem`으로 부모 컨테이너의 상단 패딩(`py-10`)을 상쇄. 헤드라인을
  3줄로 재편집하고 `text-5xl`(데스크톱 `text-6xl`)로 키워 첫 화면의
  시선을 헤드라인에 집중시킴.
- **"모임 예시" 미리보기 카드 개선**: 카드 위에 "모임 예시 — 실제
  화면 그대로예요" eyebrow 라벨(accent 점 아이콘 포함)을 추가해 이
  카드가 무엇인지 명확히 하고, 카드 상단에 얇은 accent 그린 바를
  얹어 장식 없이 밋밋하던 카드에 브랜드 포인트를 더함.
- **검증**: `tsc --noEmit`/`next build` 통과, 빌드 직후 캐시 삭제 +
  재기동. Playwright 1920px/390px 스크린샷으로 헤더 제거·히어로
  풀블리드·미리보기 카드 라벨 확인, 콘솔 에러 0건.

### 18. 히어로 안에 접어 넣었던 브랜드 마크도 완전 제거

§17에서 전역 헤더 대신 히어로 안에 작게 넣어둔 "₩ 배지 + 정산"
워드마크를 사용자가 "그냥 다 빼버려"라고 요청 — 아예 브랜드
워드마크 없이 헤드라인만 남기기로 함. `page.tsx` 히어로 상단의
브랜드 마크 블록을 삭제하고, 그만큼 히어로 상하 패딩을 다시
넓혀(`pt-14 sm:pt-20`) 헤드라인이 히어로 블록 안에서 여전히 안정적인
여백을 갖도록 조정. 검증: `tsc --noEmit`/`next build` 통과, 빌드 후
캐시 삭제+재기동, Playwright 스크린샷, 콘솔 에러 0건.
